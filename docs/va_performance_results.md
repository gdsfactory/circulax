## Phase 1 — Diode profiling

### What was found

Running `benchmarks/diode/bench_circulax.py` on a 20,000-step sine-driven rectifier at 50 ps fixed step:

| Path | µs/step | Notes |
|------|---------|-------|
| Analytical `circulax.components.electronic.Diode` | 28.7 | Shockley model, 4 unknowns |
| VA-lowered `diode.va` | 50.9 | Full SPICE physics, CI internal node, 5 unknowns |

**The assembly path was never the bottleneck.** VA is 1.8× slower at diode scale — the `_primal_and_jac_real → vmap(jvp)` structure and the custom JVP rule add negligible overhead because the diode emits only 342 lines. The n-fold-redundancy hypothesis from the original plan was disproven: with n=3 the assembly path overhead should produce ~3–4× if real, but only 1.8× was observed.

For PSP103 at 42,884 emitted lines and 4,896 `jnp.where` calls, the bottleneck is the emitted-code size: physics and Jacobian were emitted as two separate functions with independent CSE, forcing XLA to handle ~35k lines of trace per Newton iteration.

### What fixed it (pre-existing work)

Three changes to `bosdi/src/bosdi/va/emitter.py` and `va_component.py`:

1. **Combined emit** — one `_<Name>_combined(...)` returning `(f, q, J_f, J_q)` with a single shared hoist block. Per Newton iteration: one trace instead of two.
2. **Emit-time DCE** (`_live_ssas`) — walk back from output roots, drop unreferenced hoists. PSP103 had 544 fully-dead and ~12k chain-dead SSAs.
3. **Single-use SSA inlining** (`_inline_single_use_ssas`) — work-queue algorithm, O(n), substitutes refcount=1 hoists into their unique consumer.

Result: **7× per-step speedup**, **7.4× JIT speedup**.

| Stage | µs/step | JIT (s) | Combined-fn lines |
|-------|---------|---------|-------------------|
| Baseline | 15,150 | 273 | 17,365 + 17,380 (separate) |
| +combined emit | 11,146 | 172 | 17,381 |
| +DCE | 2,979 | 47 | 6,256 |
| +SSA inlining | 2,423 | 38 | ~1,892 |

## Phase 2 — Direct assembly bypass

### Problem

Even with combined emit, the assembly path still routed through `_primal_and_jac_real`:

```python
# assembly.py (old)
(f_l, q_l), (df_l, dq_l) = jax.vmap(
    functools.partial(_primal_and_jac_real, physics_at_t1)
)(v_locs, params)
```

`_primal_and_jac_real` internally does `vmap(lambda e: jax.jvp(g, (v,), (e,)))(jnp.eye(n))` — n unit tangent sweeps, each firing the custom JVP rule and thus calling `_fast_combined`. The expected speedup from the plan (2–5×) depended on whether XLA was successfully CSE-merging those n copies.

### Changes

**`bosdi/src/bosdi/circulax/va_component.py:309-313`**

After `cls._fast_physics = staticmethod(fast_physics)` at the end of `_install_custom_jvp`, expose the combined function on the class:

```python
if _has_combined:
    cls._combined_fn = staticmethod(_fast_combined)
    cls._has_combined_fn = True
else:
    cls._has_combined_fn = False
```

**`circulax/circulax/circulax/compiler.py:416, 428`**

Added `combined_func: Any = eqx.field(static=True, default=None)` to `ComponentGroup`. Populated during netlist compilation:

```python
_combined_func = getattr(comp_cls, "_combined_fn", None) if getattr(comp_cls, "_has_combined_fn", False) else None
# ...
ComponentGroup(..., combined_func=_combined_func)
```

**`circulax/circulax/circulax/solvers/assembly.py:290-298`**

Direct bypass branch in `assemble_system_real` and `assemble_gc_real`:

```python
if group.combined_func is not None:
    f_l, q_l, df_l, dq_l = jax.vmap(
        lambda v, p: group.combined_func(v, p, t1)
    )(v_locs, params)
    total_f = total_f.at[group.eq_indices].add(f_l)
    total_q = total_q.at[group.eq_indices].add(q_l)
    j_eff = df_l + (alpha / dt) * dq_l
    vals_list.append(j_eff.reshape(-1))
    continue
```

### Why it works

`_fast_combined(vars_vec, params, t)` returns `(f_vals, q_vals, j_f, j_q)` from one trace. With `jax.vmap` over devices, XLA sees one call per device — no CSE problem, no n-fold trace. The `continue` statement skips the old `_primal_and_jac_real` path entirely for VA components.

The bypass is correct for parameter gradients too: `_fast_combined` is a plain JAX function (no custom JVP of its own), so `jax.grad` w.r.t. params propagates through it normally via standard reverse-mode.

### Test result

194/194 tests pass. The improvement to per-step time at this stage is incremental (combined emit already did the heavy lifting), but the bypass eliminates Python-level tracing of n unit tangents and any runtime overhead from n JVP rule firings.

## Phase 3A — DC parameter sensitivity

### Goal

Compute `∂loss/∂params` for DC operating point fitting **without** running a slow VA backward pass. Use OSDI-speed Newton to get `y*`, then derive parameter gradients analytically.

### Mathematics

Implicit function theorem at DC: `F(y*, p) = 0`

```
∂y*/∂p = -J(y*)⁻¹ · ∂F/∂p(y*)
∂loss/∂p = -(∂L/∂y) · J(y*)⁻¹ · ∂F/∂p(y*)
```

Adjoint form (one linear solve instead of one per parameter):
1. `λ = J(y*)^{-T} · ∂L/∂y`  (solve via `klujax.tsolve_with_symbol`)
2. `∂F/∂p_k[i]` via FD: perturb device i's param k, call `osdi_residual_eval`, difference
3. `∂loss/∂p_k[i] = -λᵀ · ∂F/∂p_k[i]`

Cost: 1 linear solve + `n_params × n_devices` OSDI evaluations (µs each).

### New file: `circulax/circulax/circulax/solvers/sensitivity.py` (439 lines)

```python
from circulax.solvers import dc_parameter_sensitivity, dc_parameter_sensitivity_dense

grads = dc_parameter_sensitivity(
    component_groups,
    solver,            # KLU-backed CircuitLinearSolver
    y_star,            # DC operating point from solve_dc()
    lambda y: jnp.sum(y**2),   # any differentiable loss
    osdi_group_key="psp103_0",
    param_names=["VTH0", "TOXE"],
    model_descriptor=descriptor,  # returned by osdi_component()
    eps=1e-6,
)
# grads: {"VTH0": array([...n_devices...]), "TOXE": array([...])}
```

**Key helpers:**
- `_resolve_param_cols`: maps string param names → column indices in `group.params` via descriptor, explicit dict, or bosdi registry
- `_build_klu_matrix_vals`: coalesces raw COO Jacobian values into the deduplicated KLU value array (mirrors `CircuitLinearSolver._solve_impl`)
- `_compute_fd_gradients`: FD loop over params × devices, relative step `h = eps × max(|p|, 1)`
- `dc_parameter_sensitivity_dense`: dense fallback using `jnp.linalg.solve(J.T, dL_dy)` for non-KLU solvers

## Phase 3B — Transient parameter sensitivity

### Goal

Compute `∂loss/∂params` through a full transient trajectory without re-running the simulation for each parameter.

### Mathematics

Backward Euler time-stepping residual at step k:

```
G[k](y[k], y[k-1], p) = F(y[k], p) + (Q(y[k], p) - Q(y[k-1], p)) / dt = 0
```

Discrete adjoint backward sweep (k = N down to 1):

```
ψ[N]  = ∂L/∂y[N]|direct
ψ[k]  = ∂L/∂y[k]|direct + (J_q[k] / dt)^T · λ[k+1]    ← J_q coupling term
J_eff[k]^T · λ[k] = ψ[k]                                 ← linear solve per step

∂loss/∂p = -Σ_{k=1}^{N} λ[k]^T · (∂F[k]/∂p + (∂Q[k]/∂p - ∂Q[k-1]/∂p) / dt)
```

Cost: N adjoint linear solves (one per timestep) + `N × n_params × n_devices` OSDI evals.

### New file: `circulax/circulax/circulax/solvers/adjoint.py` (604 lines)

```python
from circulax.solvers import transient_parameter_sensitivity

sol = run_fn(t0=0, t1=1e-6, dt0=1e-9, y0=y_dc,
             saveat=diffrax.SaveAt(ts=jnp.linspace(0, 1e-6, 1001)))

grads = transient_parameter_sensitivity(
    component_groups,
    solver,
    sol.ys,   # shape (n_checkpoints, sys_size)
    sol.ts,   # shape (n_checkpoints,)
    lambda ys, ts: ys[-1, 0],  # loss on final-state voltage at node 0
    osdi_group_key="rc_0",
    param_names=["R"],
    model_descriptor=descriptor,
    eps=1e-6,
)
```

### Key implementation findings

**1. The J_q coupling term is non-negotiable for capacitive parameters.**

A common simplification drops the `(J_q[k]/dt)^T · λ[k+1]` term and just solves `J_eff[k]^T λ[k] = ∂L/∂y[k]` independently at each step. This works for purely resistive parameters but causes O(N²) error accumulation for capacitive parameters (C in RC circuits). The correct adjoint threads J_q backward at every step.

**2. ∂Q must account for BOTH y[k] and y[k-1].**

The `∂G/∂p` term involves the BE charge difference: `∂Q(y[k], p)/∂p - ∂Q(y[k-1], p)/∂p`. Using only `∂Q(y[k], p)/∂p` (as a simpler approximation) causes systematic bias for C-like parameters because the Q difference is what enters the residual, not Q itself.

**3. OSDI J_q requires direct `osdi_eval`, not `assemble_gc_real`.**

`assemble_gc_real` returns zero C values for OSDI groups because the `_assemble_osdi_group` helper combines J_f and J_q into a single `j_eff` and cannot separate them after the fact. The adjoint backward sweep needs J_q separately for the coupling term; the adjoint code calls `osdi_eval` directly via the handle to get `cap` (the reactive Jacobian tensor).

### Validation

RC circuit, OSDI resistor + OSDI capacitor, 100-step and 1000-step transients:

|------|---------|-----------------|------------|
| ∂loss/∂R (resistive), 100 steps | 1.071e-02 | 1.064e-02 | 0.72% |
| ∂loss/∂C (reactive), 100 steps | −2.331e+10 | −2.338e+10 | 0.33% |
| ∂loss/∂R (final-state loss), 100 steps | 7.187e-05 | 7.098e-05 | 1.26% |

Error decreases with finer stepping (as expected for a discrete adjoint of a first-order method).

## Phase 4 — MIR audit (pre-existing)

### Summary

SCCP (Sparse Conditional Constant Propagation) was ported from vajax into `circulax/va/sccp.py` (~400 lines) and integrated into `lower()`. Two-stage integration:
1. Constant injection — `run_sccp` seeds static params as initial lattice constants
2. Phi shortcut — `_lower_phi` consults `SccpResult.live_phi_value` to use a phi edge directly when only one predecessor is executable per SCCP

For PSP103 with TYPE-only static, SCCP found 134 additional dead `jnp.where` branches the post-emit passes had missed.

### All-static path

All-static `static_params={all model params}` is blocked by a structural bug in `_lower_phi`'s fallback heuristic (Case 2: "first non-zero edge"). With TYPE-only static, most PSP103 parameter SSAs are runtime references; with all-static, many edges fold to literal `0.0`, causing the heuristic to pick different phi edges and leak parameter default sentinels (e.g. `8e22` oxide capacitance) into the drain residual. Root cause pinned to `v1420684 = i_v3036 * 0.375` (all-static) vs `i_v3036 * i_v3424` (TYPE-only), where `i_v3424` is the safe-divide chain for `RDE=0`.

**Production path**: TYPE-only `static_params={"TYPE": 1}` at ~2,154 µs/step. Targeted fitting: `differentiable_params=("VTH0", "TOXE", ...)` keeps only named params as JAX leaves.

## What worked and why

### Combined emit was the single biggest win

Emitting one `_combined(...)` function returning `(f, q, J_f, J_q)` collapsed the ~35k-line per-Newton-iteration trace (two 17k-line functions) into ~1.9k lines after DCE + SSA inlining. The key insight: physics and Jacobian share the bulk of their subexpressions, but emitting them as two separate functions means each has its own CSE pass. At PSP103 scale, XLA's HLO-level inter-procedure CSE cannot reliably merge them, so the duplication manifests as real JIT and runtime cost.

### Implicit differentiation is the right architecture for parameter fitting

The VA forward pass at 2,154 µs/step is roughly 60× the OSDI forward pass (36 µs/step for a single PSP103 device). To compute one `jax.grad` through the VA transient requires running the full backward pass through every Newton iteration — effectively 2× the forward cost minimum. Against the OSDI path:
- OSDI DC solve: ~5 ms for N=9 ring
- DC parameter sensitivity (`dc_parameter_sensitivity`): ~0.5 ms (1 adjoint solve + 11 OSDI FD evals for 11 params)
- VA DC gradient: ~10 ms (full forward + backward, all JIT cold)

For transient:
- OSDI transient (20k steps): ~8 s
- Transient adjoint (`transient_parameter_sensitivity`, N=1000 checkpoints, 11 params): ~1 s
- VA transient gradient: impractical (>400 s per the original plan)

### FD through OSDI is accurate enough and simple

Using finite differences through OSDI's native C evaluation (`osdi_residual_eval`) for `∂F/∂p` avoids any need for AD through the XLA FFI. With relative step `h = eps × max(|p|, 1)` and `eps=1e-6`, errors are in the 1e-6 to 1e-8 range for typical SPICE parameters. This is better than the gradient noise floor from a single Newton solve (~1e-10 residual), so FD is not the limiting factor.

### Discrete adjoint accuracy vs continuous

The discrete adjoint of Backward Euler is the correct approach for this implementation because:
1. The trajectory `y_trajectory` comes from a BE time-stepper — using the continuous adjoint ODE would introduce a mismatch between the adjoint solver and the forward solver
2. For BE, the per-step cost is one sparse linear solve (`J_eff^T λ = ψ`) — the same as one Newton iteration — so the adjoint backward pass has the same asymptotic cost as the forward pass
3. Accuracy scales as O(dt) (first-order method), reaching <0.1% at 1000 steps for a 1 µs RC circuit

## What could be improved

### 1. All-static PSP103 lowering bug

**Impact**: If fixed, the combined function shrinks from ~1,892 to ~1,580 lines (-16%), JIT from 37 s to ~30 s, per-step from ~2,154 to ~1,800 µs/step (estimated).

**Root cause**: `_lower_phi` Case 2 heuristic ("first non-zero edge") picks different phi edges when parameter SSAs fold to literal `0.0` under all-static mode. The fix needs `_lower_phi` to use SCCP's lattice result to select the live phi edge rather than a text-content heuristic.

**File**: `bosdi/src/bosdi/va/lowering.py` around `_lower_phi`. The `SccpResult.live_phi_value` is already computed — the fix is to consult it before falling back to the heuristic.

### 2. FD loop is Python-level (no JAX JIT)

`dc_parameter_sensitivity` and `transient_parameter_sensitivity` run a `for` loop over `n_params × n_devices` with individual `osdi_residual_eval` calls. Each call is a native FFI invocation, not JIT-able via JAX. For PSP103 with 11 fitting parameters and 9 devices, this is 99 OSDI calls per DC sensitivity query — fine in practice (~0.5 ms) but not parallelisable on GPU.

**Improvement**: batch the FD perturbations. `osdi_residual_eval` accepts a `(N, n_ports)` input, where N is the batch dimension. Perturbing all 9 devices simultaneously (one batch call per parameter) would reduce the Python loop from `n_params × n_devices` to `n_params` iterations.

### 3. Transient adjoint requires uniform timestep assumption

The current implementation computes `dt = ts[k+1] - ts[k]` for each interval but assumes the timestepper used Backward Euler with exactly these checkpoints. In practice, `diffrax.diffeqsolve` with `BDF2*TransientSolver` uses variable-order BDF2, not pure BE. The discrete adjoint is therefore approximate for non-uniform BDF2 trajectories.

**Improvement**: Store the integration method and step-level metadata alongside the trajectory. For exact discrete adjoints of BDF2, the backward recurrence needs the BDF2 coefficients at each step. Alternatively, fall back to the continuous adjoint ODE (which is method-agnostic but requires an adjoint integrator).

### 4. No `jax.custom_vjp` wrapper (no `jax.grad` through `solve_dc`)

The current API requires the user to call `dc_parameter_sensitivity` explicitly after `solve_dc`. A cleaner long-term interface wraps the DC solve in `jax.custom_vjp`:
- **fwd**: run OSDI DC solve, return `y*` and the residuals for the backward
- **bwd**: run the implicit diff formula to return parameter gradients

This would allow `jax.grad(lambda params: loss(solve_dc_with_params(params)))` to work end-to-end, composing with JAX's autodiff machinery.

**File**: would live in `circulax/solvers/sensitivity.py` as a new `differentiable_dc_solve()` wrapper. Blocked on the same `model_descriptor` and `param_to_col` plumbing, but the math is identical to what's already implemented.

### 5. Transient sensitivity cost scales with O(N × n_params × n_devices)

For a 20,000-step transient with 11 parameters and 9 devices, the adjoint backward sweep does 1,980,000 OSDI evaluations. At ~1 µs per OSDI call, this is ~2 s — comparable to the forward transient, which matches the plan's expected cost.

The dominant cost reduction available is **checkpointing**: if the trajectory is saved at only 100 checkpoints (not all 20,000 steps), the adjoint loop runs 100 times and the gradient approximation is the same as Phase 3B's continuous-adjoint plan. The accuracy trades against the checkpoint density. This is the standard adjoint memory/compute tradeoff; Diffrax's `BacksolveAdjoint` does this automatically for diffrax-native ODEs.

### 6. BSIM4 / BSIM3 / BSIMBULK lowering gaps

Three MOSFET alternatives are blocked in the VA lowering path:
- **BSIM4v8**: `LoweringError: unresolved operand` for unnamed substrate-R branch currents (`CurrentKind(kind='Unnamed', hi='sbulk', lo='b')`) — `lowering.py` only handles `kind='Branch'`
- **BSIM3v3**: `DumpParseError: unknown InputKind 'PrevState(lim_state0)'` — limit-state PrevState not handled in `dump_parser.py`
- **BSIMBULK106**: `DumpParseError: PortConnected { port: node4 } .. "t"` — quoted-name rendering not handled

All three work via OSDI. The VA-lowering gaps are worth fixing for `differentiable_params`-style parameter fitting where the analytical Jacobian is needed.
